🍎 Setup prostředí – MacBook Pro M3 Max, 36 GB

Homebrew, pyenv, Python 3.12, Cursor, Ollama, MLX – krok za krokem

▶
  • 1Nainstaluj Homebrew – správce balíčků pro Mac. Otevři Terminal (Cmd+Space → „Terminal") a spusť:
    /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
    ⚠️ Na M3 (Apple Silicon) je Homebrew v /opt/homebrew/ – ne v /usr/local/ jako na Intel Macu. Po instalaci přidej do PATH – installer ti to řekne přesně. Typicky: echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> ~/.zshrc && source ~/.zshrc
    Dokumentace: brew.sh
  • 2Nainstaluj pyenv – správce verzí Pythonu:
    brew install pyenv
    Přidej do ~/.zshrc tyto tři řádky:
    export PYENV_ROOT="$HOME/.pyenv"
    export PATH="$PYENV_ROOT/bin:$PATH"
    eval "$(pyenv init -)"
    Pak: source ~/.zshrc
    Dokumentace: github.com/pyenv/pyenv
  • 3Nainstaluj Python 3.12:
    pyenv install 3.12.5 && pyenv global 3.12.5
    Ověř: python --version → Python 3.12.5
    ✅ Na M3 Max se Python kompiluje nativně pro ARM – je výrazně rychlejší než na Intel Macu. PyTorch i NumPy využijí plný výkon čipu.
  • 4Nainstaluj Cursor (doporučuji nad VS Code pro AI development):
    cursor.com → Download → vyber Apple Silicon verzi → přetáhni do Applications
    Cursor = VS Code + nativní AI asistence. Všechny VS Code extensions fungují stejně. Cmd+Shift+P → „Shell Command: Install 'cursor' command in PATH".
    Alternativa: VS Code → taktéž stáhni Apple Silicon verzi.
  • 5Pro každý projekt vytvoř virtuální prostředí:
    mkdir ai-projects && cd ai-projects
    python -m venv .venv
    source .venv/bin/activate ← spouštíš pokaždé při otevření nového Terminálu
    Vidíš (.venv) před promptem? Prostředí je aktivní.
  • 6Nainstaluj JupyterLab a základní balíčky:
    pip install jupyterlab numpy matplotlib pandas python-dotenv
    Spusť: jupyter lab → otevře se na http://localhost:8888
  • 7Vytvoř .env pro API klíče:
    touch .env && echo ".env" >> .gitignore
    Obsah .env: OPENAI_API_KEY=sk-... · ANTHROPIC_API_KEY=sk-ant-...
    Nikdy nepushe .env na GitHub.
  • 8🚀 M3 Max speciál: nainstaluj Ollama – spouštění LLM lokálně, zdarma, offline:
    Stáhni z ollama.com → přetáhni do Applications. Pak:
    ollama pull llama3.2 → Llama 3.2 3B (~2 GB, rychlý, pro experimenty)
    ollama pull mistral → Mistral 7B (~4 GB, doporučený sweet spot)
    ollama pull llama3.1:8b → Llama 3.1 8B (~5 GB, výborný výkon)
    ollama pull phi3:14b → Phi-3 Medium 14B (~8 GB, silný model)
    ollama run mistral → spustí chat v Terminálu
    ⚠️ Llama 3.1 70B nedoporučuji: v 4-bit kvantizaci potřebuje ~40 GB, macOS systém spotřebuje dalších ~4–6 GB – na 36 GB to nepůjde plynule. Reálný strop je ~20–24 GB modely (např. Qwen 2.5 14B nebo Gemma 2 27B). Mistral 7B a Llama 3.1 8B jsou praktický sweet spot pro tento kurz.
    Ollama poskytuje OpenAI-kompatibilní API na localhost:11434 – použij místo OpenAI API při vývoji a šetři náklady.
  • 9🚀 M3 Max speciál: nainstaluj MLX – Apple framework optimalizovaný pro M-čipy:
    pip install mlx mlx-lm
    MLX využívá Apple Silicon naplno – je až 2× rychlejší než PyTorch pro inferenci na stejném hardwaru.
    Test – spusť Mistral 7B lokálně:
    mlx_lm.generate --model mlx-community/Mistral-7B-Instruct-v0.3-4bit --prompt "Co je RAG systém?"
    Model se stáhne jednou (~4 GB), pak běží offline. Na M3 Max generuje ~60–80 tokenů/sec.
    Dokumentace: ml-explore.github.io/mlx · Modely: huggingface.co/mlx-community
🎯 Co ti dává M3 Max (36 GB) oproti ostatním Unified memory = GPU i CPU sdílí stejných 36 GB. Na běžném notebooku jsou RAM a GPU VRAM oddělené, takže modely narážejí na limity dříve. Na M3 Max máš výrazně větší prostor – ale macOS samo spotřebovává ~4–6 GB, takže efektivní limit pro modely je ~28–30 GB.
  • Mistral 7B, Llama 3.1 8B, Gemma 2 9B – ideální sweet spot, běží plynule v 4-bit kvantizaci (~4–6 GB), offline, zdarma
  • Phi-3 Medium 14B, Qwen 2.5 14B – silnější modely (~8–9 GB), stále rychlé a pohodlné
  • Gemma 2 27B – velký model (~16 GB v 4bit), na 36 GB funguje, pomalejší (~15–25 tok/sec)
  • ⚠️ 70B modely nedoporučuji – v 4-bit kvantizaci potřebují ~40 GB, systém nestačí; modely nad ~28 GB budou swapovat na disk a generace bude nepoužitelně pomalá
  • Fine-tuning malých modelů (LoRA na GPT-2, FLAN-T5) přes MLX přímo na Macu – nepotřebuješ platit Colab GPU
  • Embeddingy a inference HuggingFace modelů (distilbert, sentence-transformers) lokálně, instantně
  • Při vývoji agentů (L14–L15) používej Ollama jako backend místo OpenAI API – šetříš stovky Kč na API nákladech

🚨 Troubleshooting – co dělat když něco nefunguje

ImportError, NaN loss, API chyby, Ollama, Jupyter kernel – rychlá řešení

▶
ModuleNotFoundError / ImportError ModuleNotFoundError: No module named 'openai'
Příčina: Balíček není nainstalovaný ve tvém aktivním venv.
Řešení: 1) Zkontroluj, že je venv aktivní – vidíš (.venv) v promptu? 2) Spusť pip install openai. 3) Pokud používáš Jupyter, restartuj kernel (Kernel → Restart).
Loss je NaN nebo Inf (L2, L3) Loss: nan nebo Loss: inf po pár krocích trénování.
Příčina: Learning rate je příliš velký – váhy explodují.
Řešení: Zmenši learning rate 10×. Např. lr=0.1 → lr=0.01. Pokud stále NaN, zkus inicializovat váhy * 0.01 místo * 0.1.
OpenAI API chyba – AuthenticationError AuthenticationError: No API key provided
Příčina: API klíč není načtený.
Řešení: 1) Zkontroluj, že .env soubor existuje a obsahuje OPENAI_API_KEY=sk-.... 2) Ujisti se, že voláš load_dotenv() před prvním API voláním. 3) Ověř: import os; print(os.getenv("OPENAI_API_KEY")) – nesmí být None.
OpenAI API – RateLimitError RateLimitError: You exceeded your current quota
Příčina: Buď nemáš kredit na účtu, nebo přesahuješ limity volání.
Řešení: 1) Zkontroluj kredit na platform.openai.com/usage. 2) Pro experimenty používej gpt-4.1-nano (nejlevnější) nebo lokální Ollama (zdarma).
Ollama – connection refused (L11+) ConnectionError: http://localhost:11434 – connection refused
Příčina: Ollama aplikace neběží.
Řešení: Otevři aplikaci Ollama (Applications → Ollama) nebo spusť ollama serve v Terminálu. Ověř: curl http://localhost:11434/api/tags – mělo by vrátit JSON se seznamem modelů.
Jupyter kernel se zasekl nebo nereaguje Příčina: Nekonečná smyčka, velký výpočet, nebo zaseklý API call.
Řešení: 1) Zmáčkni ■ Stop (čtvereček vedle buňky) nebo Kernel → Interrupt. 2) Pokud nereaguje: Kernel → Restart Kernel. 3) Po restartu musíš znovu spustit všechny buňky od začátku (Run → Run All Cells Above).
FAISS instalace selhala na Macu (L12) ERROR: Could not build wheels for faiss-cpu
Řešení: Na Apple Silicon Macu použij: pip install faiss-cpu --no-binary faiss-cpu nebo alternativně pip install chromadb (ChromaDB funguje na M3 bez problémů a je dobrá náhrada za FAISS).
Obecná strategie při jakékoliv chybě 1. Přečti celou chybovou hlášku – poslední řádek je nejdůležitější.
2. Zkopíruj chybu do Cursoru (Cmd+L) a zeptej se "co tato chyba znamená a jak ji opravím?"
3. Hledej na Stack Overflow nebo GitHub Issues – 95 % chyb někdo řešil před tebou.
4. Izoluj problém – zmenši kód na nejmenší možný příklad který stále chybuje.

📓 JupyterLab průvodce

Buňky, klávesové zkratky, kernel, grafy, instalace balíčků

▶

JupyterLab je interaktivní prostředí v prohlížeči, kde píšeš kód po malých blocích a vidíš výsledky okamžitě – ideální pro experimenty s AI. Dokumentace: jupyterlab.readthedocs.io

SpuštěníV Terminálu (s aktivním venv): jupyter lab → otevře se prohlížeč. Soubory notebooků mají příponu .ipynb. Nový notebook: File → New → Notebook.
BuňkyNotebook = série buněk. Buňka může být kód (Python) nebo Markdown (text). Přepínání: nahoře vybereš Code nebo Markdown.
Spuštění kóduShift+Enter spustí buňku a přejde na další. Ctrl+Enter spustí buňku a zůstane na ní. Výsledek se zobrazí hned pod buňkou.
Klávesové zkratkyA = nová buňka nad, B = nová buňka pod, DD = smaž buňku, M = změň na Markdown, Y = změň na kód. (Tyto fungují v Command mode – klikni vedle buňky, ne do ní.)
KernelKernel = Python proces který běží v pozadí. Pokud se notebook zasekne: Kernel → Restart. Proměnné po restartu zmizí – musíš spustit buňky znovu od začátku.
Výstup a grafyGrafy (Matplotlib, Seaborn) se zobrazují přímo v notebooku. Stačí zavolat plt.show() nebo jen napsat jméno proměnné jako poslední řádek buňky.
Instalace balíčkůV notebooku piš !pip install název (vykřičník = shell příkaz). Tím nainstaluješ přímo do aktivního venv bez opuštění notebooku.
UloženíCmd+S uloží notebook. Jupyter autosave každé 2 minuty. Soubor .ipynb je vlastně JSON – dá se verzovat v Gitu (ale je to trochu chaotické).

📦 Virtual environments a pip ZDARMA
Každý projekt má vlastní venv – izolované prostředí s vlastními balíčky. Nikdy nemíchej balíčky napříč projekty.

Workflow pro každý nový projekt:
mkdir muj-projekt && cd muj-projekt
python -m venv .venv
source .venv/bin/activate ← spouštíš pokaždé
Vidíš (.venv) v promptu? Aktivní.
pip install balicek ← instaluje jen do .venv
pip freeze > requirements.txt ← seznam závislostí

🔐 Správa API klíčů a .env BEZPEČNOST

Kde získat klíče:

⚡ uv – rychlý správce balíčků ZDARMA
uv je náhrada za pip napsaná v Rustu – je 10–100× rychlejší a automaticky spravuje venv za tebe. V 2026 se stává standardem pro Python projekty.

Instalace:
brew install uv

Základní příkazy (místo pip + venv):
  • uv init muj-projekt – vytvoří projekt s pyproject.toml
  • uv add openai langchain – přidá balíčky (automaticky vytvoří venv)
  • uv run python skript.py – spustí skript ve správném venv
  • uv sync – nainstaluje vše z pyproject.toml (jako pip install -r)
  • uv pip install balicek – klasický pip-kompatibilní příkaz
Pro kurz: Používej buď klasický venv + pip (jak je popsáno výše) nebo uv – oba přístupy fungují. uv doporučuji pro nové projekty od L11 dál.

    💻 Kódování Dvouvrstvá síť – forward pass
    Proč to děláš

    Jeden neuron nestačí na složité problémy. Teď postavíš síť ze dvou vrstev – přesně tak jak to vypadá ve videu. Uvidíš, že i celá síť je jen maticové násobení. Výsledky budou zatím špatné – to je v pořádku, trénování přidáme v L2.

    XOR problém – co budeme řešit

    Chceme síť která řekne: "jsou tato dvě čísla různá?" Vstup [0,1] → odpověď 1 (ano, různá). Vstup [1,1] → odpověď 0 (ne, stejná). Tenhle problém nejde vyřešit jednou přímkou – proto potřebujeme dvě vrstvy.

    Kód — pokračuj ve stejném notebooku, nová buňka
    # Vstupní data: 4 kombinace dvou bitů # Každý řádek = jeden příklad X = np.array([[0,0], [0,1], [1,0], [1,1]]) # Správné odpovědi pro XOR: "jsou různé?" y = np.array([[0], # 0,0 → stejné → 0 [1], # 0,1 → různé → 1 [1], # 1,0 → různé → 1 [0]]) # 1,1 → stejné → 0 # Váhy první vrstvy: 2 vstupy → 4 neurony # randn = náhodná čísla z normálního rozdělení # * 0.1 = malé hodnoty, aby sigmoid nezasytil hned W1 = np.random.randn(2, 4) * 0.1 # Váhy druhé vrstvy: 4 neurony → 1 výstup W2 = np.random.randn(4, 1) * 0.1 # Sigmoid funkce – stejná jako v předchozí buňce def sigmoid(x): return 1 / (1 + np.exp(-x)) # FORWARD PASS: data tečou sítí zleva doprava # @ = maticové násobení (jako np.dot ale čitelnější) h = sigmoid(X @ W1) # výstup první vrstvy, tvar: (4, 4) y_hat = sigmoid(h @ W2) # výstup celé sítě, tvar: (4, 1) print("Predikce PŘED trénováním:") print(y_hat.T.round(2)) print("\nSprávné odpovědi:") print(y.T)

    Predikce budou náhodná čísla blízko 0.5 – síť ještě nic neví, váhy jsou náhodné. Je to normální. V L2 přidáme trénování a síť se naučí správné odpovědi.

    💻 Kódování Trénování XOR sítě – loss klesá
    Proč to děláš

    Síť z L1 měla náhodné váhy a špatné predikce. Teď ji naučíš správné odpovědi. Uvidíš číslo "loss" – chybu sítě – jak postupně klesá. Tohle je doslova to samé co se děje při trénování GPT, jen ve sto milionkrát menším měřítku.

    Co je loss a gradient descent

    Loss = číslo které říká "jak moc je síť špatná". Čím nižší, tím lépe. Cíl trénování: snižovat loss.
    Gradient descent = algoritmus který říká: "změň váhy tímto směrem a o tolik, aby loss klesl".
    Learning rate = jak velký krok děláme. Moc velký → přeskočíme nejlepší bod. Moc malý → trénování trvá věčně.

    Intuice gradientuPředstav si, že stojíš na kopci ve mlze a chceš sejít do údolí (= minimální chyba). Gradient ti říká, jakým směrem jde kopec nahoru – ty jdeš opačně. Learning rate určuje, jak velký krok uděláš. Moc velký krok → přeskočíš údolí. Moc malý → nikdy nedojdeš. Backpropagation je jen efektivní způsob jak spočítat tento gradient pro tisíce vah najednou pomocí řetězového pravidla derivací.
    Kód — nová buňka ve stejném notebooku (X, y, W1, W2 a sigmoid jsou stále v paměti z L1)
    lr = 0.5 # learning rate = velikost kroku for epoch in range(5000): # 1. FORWARD PASS: spočítej co síť předpovídá h = sigmoid(X @ W1) y_hat = sigmoid(h @ W2) # 2. LOSS: jak moc se mýlí? (průměrná kvadratická chyba) loss = ((y - y_hat) ** 2).mean() # Každých 500 epoch vytiskni progress if epoch % 500 == 0: print(f"Epoch {epoch:5d}: loss = {loss:.4f}") # ─── BACKWARD PASS ───────────────────────────────────────────────── # Cíl: zjistit, jak moc každá váha přispěla k chybě # Postup: jdeme od výstupu ZPÁTKY k vstupům (odtud "backpropagation") # Derivace = říká "o kolik se výstup změní, když trochu posunu tuto váhu" # Chyba výstupu – jak moc se lišíme od správné odpovědi # dL/dy_hat = derivace MSE loss; * 2 je konstanta z (y-ŷ)² dL = 2 * (y_hat - y) / len(y) # Gradient pro W2 – jak moc W2 způsobil chybu # h.T = transponovaná matice výstupů první vrstvy # * y_hat * (1 - y_hat) = derivace sigmoidu (je maximální u 0.5, nula u 0 a 1) dW2 = h.T @ (dL * y_hat * (1 - y_hat)) # Gradient pro W1 – propagujeme chybu dál dozadu přes W2 # @ dW2.T = "kolik z té chyby prišlo přes W2?" # * h * (1 - h) = znovu derivace sigmoidu pro první vrstvu dW1 = X.T @ ((dL * y_hat * (1 - y_hat)) @ dW2.T * h * (1 - h)) # UPDATE VAH – posuneme váhy ve směru OPAČNÉM k gradientu # lr = learning rate = jak velký krok uděláme (příliš velký → přeskočíme minimum) # Intuice: jdeme "do kopce" gradientu, ale obráceně → najdeme minimum chyby W2 -= lr * dW2 W1 -= lr * dW1 print("\nVýsledné predikce po trénování:") print(sigmoid(sigmoid(X @ W1) @ W2).T.round(2)) print("\nSprávné odpovědi:") print(y.T)

    Co jsou Q, K, V (Query, Key, Value)?Attention mechanismus funguje jako vyhledávání v databázi:
    • Query (Q) = "co hledám?" – otázka aktuálního tokenu
    • Key (K) = "co nabízím?" – popis každého tokenu v sekvenci
    • Value (V) = "co vrátím?" – skutečná informace tokenu

    Příklad: Věta "Banka řeky je mokrá." Pro slovo "banka" je Q="co znamenám?", K každého okolního slova říká "jsem řeka/mokrý/mokrá", V vrátí relevantní kontext. Attention skóre = jak moc Q odpovídá každému K → víme na co se zaměřit.

    MultiheadAttention dělá tento výpočet paralelně v několika "hlavách" – každá hlava se učí jiný typ vztahu (gramatický, sémantický, poziční...).

    💻 Kódování Jeden GPT blok složený z komponent
    Proč to děláš

    GPT-3 má 96 identických bloků naskládaných za sebou. Teď postavíš jeden takový blok ze tří komponent které jsi viděl ve videu: attention + normalizace + feed-forward. Pochopíš co každá část dělá a proč tam je.

    Kód — nová buňka
    class GPTBlok(nn.Module): """Jeden blok GPT transformeru – přesně co je ve videu""" def __init__(self, dimenze=64, pocet_hlav=4): super().__init__() # 1. ATTENTION – tokeny si navzájem sdělují informace self.attention = nn.MultiheadAttention(dimenze, pocet_hlav, batch_first=True) # 2. LAYER NORM – normalizuje čísla pro stabilní trénování # (bez toho by čísla "explodovala" na obrovské hodnoty) self.norm1 = nn.LayerNorm(dimenze) self.norm2 = nn.LayerNorm(dimenze) # 3. FEED-FORWARD – každý token zpracuje sám pro sebe # 4× větší střední vrstva je standard v GPT architektuře self.ff = nn.Sequential( nn.Linear(dimenze, dimenze * 4), # rozšíř nn.ReLU(), # aktivace nn.Linear(dimenze * 4, dimenze), # zúži zpět ) def forward(self, x): # Attention + residual connection (přičteme původní x) # Residual = "zkratka" aby gradienty mohly téct zpět attn_vystup, _ = self.attention(x, x, x) x = self.norm1(x + attn_vystup) # Feed-forward + residual connection x = self.norm2(x + self.ff(x)) return x # Otestujeme blok blok = GPTBlok() vstup = torch.randn(1, 10, 64) # 1 věta, 10 tokenů, 64 dimenzí vystup = blok(vstup) print(f"Vstup: {vstup.shape}") print(f"Výstup: {vystup.shape}") print("✓ Tvar se nezměnil – blok transformuje tokeny, nemění jejich počet") # Kolik parametrů (= vah) má jeden blok? pocet_parametru = sum(p.numel() for p in blok.parameters()) print(f"\nPočet parametrů v jednom bloku: {pocet_parametru:,}") print(f"GPT-3 má 96 takových bloků → {96 * pocet_parametru:,} parametrů (pouze z bloků)")
    Co uvidíš po spuštění

    Tvar vstupu a výstupu je stejný — blok transformuje tokeny ale nemění jejich počet. Uvidíš také počet parametrů. Každý parametr = jedno číslo které se učí trénováním. GPT-3 má 175 miliard takových čísel.

  • 2Generování jmen z bigram modelu
    💻 Nakóduj: Převeď bigram tabulku na pravděpodobnosti a generuj jména:
    # Normalizace: každý řádek tabulky musí součtem dávat 1.0 P = (N + 1).astype(float) # +1 = smoothing (žádná nulová pravd.) P = P / P.sum(axis=1, keepdims=True) # normalizace po řádcích def generuj(): jmeno, idx = "", 0 while True: # Vzorkuj příští znak podle pravděpodobností aktuálního znaku idx = np.random.choice(vocab, p=P[idx]) if idx == 0: break # konec slova jmeno += itos[idx] return jmeno print("Vygenerovaná jména:", [generuj() for _ in range(10)])

    from langchain_core.messages import HumanMessage, AIMessage
    from typing import TypedDict, Annotated
    import operator
    class AgentState(TypedDict):
    messages: Annotated[list, operator.add] # seznam zpráv, každý uzel přidává
    def call_llm(state: AgentState) -> AgentState:
    response = llm.bind_tools(nastroje).invoke(state["messages"])
    return {"messages": [response]}

    last = state["messages"][-1]
    return "tools" if hasattr(last, "tool_calls") and last.tool_calls else END
    from langgraph.prebuilt import ToolNode
    graph = StateGraph(AgentState)
    graph.add_node("llm", call_llm)
    graph.add_node("tools", ToolNode(nastroje))
    graph.set_entry_point("llm")
    graph.add_conditional_edges("llm", should_continue)
    graph.add_edge("tools", "llm")
    app = graph.compile()
    result = app.invoke({"messages": [HumanMessage("Co dělá JIC?")]})
    print(result["messages"][-1].content)

MCP – Model Context Protocol
  • +Co je MCP a proč ho znát
    MCP (Model Context Protocol, Anthropic 2024) je otevřený standard pro propojování AI agentů s nástroji, databázemi a API. Cursor, Claude Desktop a většina moderních AI nástrojů v 2026 používá MCP.

    Klíčové koncepty:
    • MCP Server – poskytuje nástroje (tools) a zdroje (resources) agentovi
    • MCP Client – agent který MCP servery volá (Cursor, LangGraph, Claude Desktop)
    • Transport – stdio (lokální procesy) nebo HTTP+SSE (vzdálené servery)

    Proč to musíš znát: Pokud v L17–18 nasazuješ vlastní agent, zákazníci a integrace budou čekat MCP-kompatibilní rozhraní. Je to nový REST API standard pro AI agenty.

    📖 Přečti: modelcontextprotocol.io/introduction (~15 min)
    📺 Sleduj: Jak MCP funguje v Cursoru: Cursor Settings → MCP → prohlédni dostupné servery
    TipCursor Pro+ ti dává přístup k MCP serverům přímo v IDE. Vyzkoušej: přidej Tavily MCP server do Cursoru a místo TavilySearchResults v LangGraphu použij MCP tool – je to identické z pohledu agenta.

Průvodce stavbou AI agentů – od základní architektury po produkční nasazení. Pokrývá LangGraph, AutoGen, CrewAI a multi-agent systémy s konkrétními příklady. Napsáno prakticky pro vývojáře, ne akademiky.
Proč číst: Přímá podpora pro Blok 4 kurzu (L14–L15). Přečti před nebo souběžně s L14 (LangGraph). Nejnovější kniha k tématu agentů dostupná v e-formátu.

Proč číst: Ideální jako vizuální doplněk k technickému studiu v L1–L4 (neurony, transformery) a L9–L10 (HuggingFace, embeddingy). Pokud ti nestačí text, tato kniha ti ukáže co se děje uvnitř. Kód na GitHubu (ZDARMA).
